Activitats

Agregació i composició

L’objectiu d’aquesta activitat és aprofundir en els conceptes dels diagrames de classes, concretament en els que fan referència als tipus de relacions que existeixen entre les classes i, més concretament encara, a les relacions d’associació.

Explica detalladament la diferència entre els conceptes: relació d’associació d’agregació i relació d’associació de composició. Fes servir un o diversos exemples per acompanyar les teves explicacions.

L’agregació i la composició són dos tipus de relacions d’associació entre classes.

Agregació: mostra un significat de relació dèbil entre dues classes. Semànticament, una relació d’associació de tipus agregació té el significat del verb tenir.

Per exemple, com es pot veure a la figura, es pot crear la següent relació entre dues classes:

Un curs té assignatures.

Quan s’elimina un curs, no és necessari eliminar les assignatures que el componen, ja que aquestes poden ser utilitzades per a un altre curs o per a una altra carrera universitària.

Composició: mostra un significat de relació forta entre dues classes. Semànticament, una relació d’associació de tipus composició té el significat del verb “estar compost de”.

Per exemple, com es pot veure a la figura, es pot crear la següent relació d’associació de composició entre dues classes:

Un avió es compon de 2 ales.

Quan s’elimina un avió s’han d’eliminar les seves ales, ja que tota ala formarà part de l’avió.

Un altre exemple que fa ús a la vegada de les relacions d’associació d’agregació i de composició és el que es mostra a la figura.

En aquest exemple es podrien establir els següents significats:

L’aplicació té una base de dades.

L’aplicació es compon de codi de programació.

Classe associativa

L’objectiu d’aquesta activitat és aprofundir en els conceptes dels diagrames de classes, concretament en els que fan referència al concepte de classe associativa.

A partir del següent diagrama de classes (figura):

  • Interpreta el significat del diagrama de classes que es mostra a la figura següent, indicant el significat de les classes i les seves relacions.
  • Modifica el diagrama de classes per tal de poder ser implementat en un llenguatge de programació orientat a objectes, com per exemple Java.

Quan una associació té propietats pròpies es representa com una classe unida a la línia de l’associació per mitjà d’una línia discontinua. Tant la línia com el rectangle de classe representen el mateix element conceptual: l’associació.

En el cas de l’exemple ens trobem que la nota està directament relacionada amb les classes Estudiant, Assignatura: cada un dels alumnes de l’assignatura tindrà una determinada nota. La manera que té l’UML de modelitzar aquesta situació és amb les classes associades.

D’altra banda, les classes associatives no existeixen en els llenguatges de programació, per tant, és necessari haver de dur a terme una transformació en el model de classes per tal de poder ser codificat.

La transformació consisteix a convertir la classe associativa en una classe amb un únic atribut, que estarà relacionada amb les classes Estudiant i Assignatura, tal com es mostra a la figura figura.

Classe associativa vs Classe ternària

L’objectiu d’aquesta activitat és aprofundir en els conceptes dels diagrames de classes, concretament en els que fan referència al concepte de classe associativa i de classe ternària.

Es demana donar solució a les dues qüestions següents:

  1. Expliqueu les diferències entre una classe associativa i una classe ternària. Feu servir un exemple clarificador.
  2. Representeu, d’una forma equivalent, una classe ternària.

Classe associativa

  • Una matrícula correspon a l’associació entre un estudiant i una assignatura.
  • Un estudiant pot haver-se matriculat a diverses assignatures, fins i tot un estudiant pot haver-se matriculat diverses vegades a una mateixa assignatura.
  • Una assignatura pot haver estat referenciada en la matrícula per diversos estudiants.

Relació ternària

  • Una matrícula pot estar associada diverses vegades amb estudiants i assignatures.
  • Un estudiant pot haver-se matriculat a diverses assignatures, fins i tot un estudiant pot haver-se matriculat diverses vegades a una mateixa assignatura.
  • Una assignatura pot haver estat referenciada en la matrícula per diversos estudiants.

Herència. Enginyeria inversa

L’objectiu d’aquesta activitat és aprofundir en els conceptes dels diagrames de classes, concretament en els que fan referència al concepte d’herència.

A partir del següent codi de programació, es demana la representació del diagrama de classes. Es tracta d’aplicar l’enginyeria inversa per obtenir el disseny en UML a partir de la codificació. 

  1. class FiguraGeometrica { 
  2. 	protected int amplada;
  3. 	protected int llargada;
  4.  
  5. 	public FiguraGeometrica (int a, int b) {  
  6. 		amplada=a; 
  7. 		llargada=b;
  8. 	} 
  9. }
  10.  
  11. class Rectangle extends FiguraGeometrica { 
  12. 	public int area() { 
  13. 		return (amplada * llargada); 
  14. 	} 
  15. }
  16.  
  17. class Triangle extends FiguraGeometrica { 
  18. 	public int area() { 
  19. 		return (amplada * llargada / 2); 
  20. 	}
  21. }
  22.  
  23. public static void main(String[] args) { 
  24.  
  25. 	Rectangle rectangleA; 
  26. 	rectangleA = new Rectangle(5,10);
  27. 	System.out.print("area del rectangle " + rectangleA.area());
  28.  
  29. 	Triangle triangleB;
  30. 	triangleB = new Triangle(5,10);
  31. 	System.out.print("area del triangle " + triangleB.area()); 
  32. }

El diagrama de classes corresponent al codi seria el que es mostra a la figura.

Cal fixar-se que els atributs de la classe FiguraGeometrica, amplada i llargària, tenen la visibilitat de tipus protegida.

Això significa que les classes que hereten de FiguraGeometrica, Triangle i Rectangle, tindran visibilitat sobre aquests atributs.

Diagrama de classes de gestió de projectes

L’objectiu d’aquesta activitat és crear un diagrama de classes a partir d’una presa de requeriments.

Elabora el diagrama de classes que doni solució al següent enunciat:

Es vol desenvolupar una aplicació informàtica que permeti dur a terme la gestió de diversos projectes informàtics. Caldrà tenir emmagatzemada tota la informació referent a cada projecte que es vulgui gestionar. Hi haurà dos tipus de persones que podran estar involucrades en els projectes, ja sigui per a la seva gestió o per a l’execució de les activitats. Aquests dos tipus de persones són el cap de projecte i el programador. De cada persona es voldrà emmagatzemar el seu nom i el seu DNI.

Un projecte serà liderat per un cap de projecte. Però un cap de projecte només podrà gestionar tres projectes. Un projecte té activitats; cada projecte podrà tenir des d’una a moltes (de forma indefinida) activitats, però una activitat només podrà pertànyer a un projecte.

Els programadors desenvoluparan les activitats. Cada programador tindrà vinculada una activitat com a mínim, sense un màxim establert. Igualment, les activitats podran ser desenvolupades per un mínim d’un programador, sense haver-hi un màxim establert.

Una possible representació podria ser la que es mostra a la figura.

Cal tenir present, en qualsevol cas, l’herència entre cap de projecte, programador i persona, les multiplicitats en les relacions de les classes i les relacions d’agregació que pugui haver-hi.
